## 1. 文档概述

### 1.1 背景

V1 已完成首页 / 组合总览等基础能力，因此 V2 不再重复建设首页主能力，而是在现有账户看板基础上新增“持仓分析”主模块，补齐“解释风险、解释变化、解释影响”的分析层。

当前用户需求已明确包含以下方向：

1. 增加持仓分析板块
2. 接入 Telegram，用于日报推送与快速查询
3. 上线只读 MCP Server，供外部 AI 客户端查询账户分析结果
4. 使用 ECharts 代替 SVG
5. 使用 Nothing Style 重绘前端 UI 风格
6. 在持仓分析内整合市场分析与个股分析能力

### 1.2 问题定义

当前系统主要回答“我持有哪些资产”，但还不能稳定回答：

- 我的组合真正暴露在什么风险下？
- 当前市场状态会怎样影响我的持仓？
- 当前持仓个股最近发生了什么，以及为什么重要？
- 接下来我最该关注什么？

### 1.3 本期目标

V2 的目标是把 `ibkr-dashboard` 从“账户数据看板”升级为“以持仓为中心的 AI 分析系统”，重点提供：

1. 当前组合风险解释
2. 与持仓相关的市场分析
3. 与持仓相关的个股分析
4. 风险预警与日报触达
5. 面向外部 AI 的只读分析接口

### 1.4 本期约束

1. **不涉及期权相关内容**，Greeks、到期风险、期权策略放到下一版
2. **首页 / 组合总览沿用 V1**，本期不重复建设首页主能力
3. **AI Summary 不放首页**，统一放入“持仓分析”模块中
4. **不把 Futu OpenD 作为基础依赖**
5. **不提供交易功能**，包括下单、改单、撤单、策略执行

## 2. 产品定位

### 2.1 一句话定位

一个围绕当前持仓展开的本地只读 AI 风险理解与决策辅助系统。

### 2.2 产品边界

本产品不做独立的市场资讯平台，也不做独立的个股研究终端。

市场分析与个股分析均嵌入“持仓分析”模块，仅服务于以下问题：

- 当前市场状态如何影响我的组合？
- 当前持仓个股最近发生了什么？
- 这些变化会不会改变我的风险结构？

### 2.3 非目标

1. 自动交易
2. 期权分析与期权策略
3. 独立行情终端
4. 回测系统
5. 多账户聚合管理
6. 把 Futu OpenD 作为强依赖基础设施

## 5. 设计原则

1. **以持仓为中心**：所有分析必须围绕当前持仓展开
2. **解释优先**：不只展示数据，还要解释“这意味着什么”
3. **只读优先**：Telegram 与 MCP 默认只读，避免误操作
4. **弱依赖外部源**：系统核心能力必须在不依赖 Futu OpenD 的情况下成立
5. **图表为理解服务**：图表用于解释结构、风险、变化，不为炫技
6. **风格统一**：整体视觉采用 Nothing Style 方向，强调简洁、克制、网格秩序、轻科技感与高对比信息层级，但不牺牲金融数据可读性

## 8. 功能需求

### 8.1 持仓分析

#### 8.1.1 模块目标

“持仓分析”是 V2 的核心模块，负责把市场分析、个股分析、组合分析统一起来，回答：

- 当前组合最核心的风险是什么？
- 当前市场状态会如何影响这些持仓？
- 当前持仓个股最近发生了什么，以及为什么重要？

#### 8.1.2 AI Summary

**功能说明**：AI Summary 从首页移入持仓分析页，作为该页的统一入口摘要。

**输出目标**：用 3 到 6 条简洁语言概括：
1. 当前组合主要暴露方向
2. 今日最重要变化
3. 当前最大风险
4. 最值得关注的后续信号

**示例**：
- 当前组合为高成长、高 AI beta 暴露组合。
- 今日风险主要来自半导体与 AI 基础设施链条的同步波动。
- 当前隐藏风险是组合高度依赖 hyperscaler AI CapEx 预期延续。
- 未来 48 小时需重点关注持仓相关财报与宏观数据。

#### 8.1.3 组合风险

**功能说明**：解释当前组合真正暴露的风险，而不是只显示仓位明细。

**分析维度**：
1. 因子暴露
2. 集中度分析
3. 相关性分析
4. 尾部风险
5. AI 主题暴露
6. 宏观敏感性

**输入数据**：
1. 持仓明细
2. 历史收益序列
3. 行业分类
4. 主题标签
5. 外部价格与事件数据

**输出内容**：
1. 当前组合属于什么类型暴露
2. 核心风险因子有哪些
3. 哪些仓位表面分散、实际高度相关
4. 当前最需要警惕的单一变量是什么

**图表建议**：
1. 风险暴露条形图
2. 相关性热力图
3. 行业集中度图
4. 主题暴露图

#### 8.1.4 持仓关联市场分析

**功能说明**：市场分析不独立存在，而是专门解释“当前市场状态会如何影响我的持仓”。

**数据维度**：你现有草稿中已经定义了这组市场分析指标，可作为该模块的候选维度：评论情绪、RSI、IV percentile、Put/Call、新闻热度、成交量异常、ETF 流入、分析师 revision。

**状态标签（保留现有设计）**：
1. Fear Compression
2. Capitulation Zone
3. Euphoric Momentum
4. Crowded Long
5. Narrative Exhaustion 

**本期实现原则**：
1. 本模块优先使用通用行情数据源、新闻与公告抓取、分析师预期变动数据和本地规则引擎
2. Futu Skills Hub 用于补充散户情绪、市场新闻、评论情绪等增强信号
3. Futu OpenD 主要用于提供更完整的行情与交易侧数据，若未接入，不影响本模块核心分析能力
4. 所有外部提供方必须通过 provider 抽象层接入，可替换、可降级

**输出目标**：
1. 当前市场状态标签
2. 该状态对当前组合的影响
3. 需要重点关注的持仓方向
4. 今日组合机会 / 风险

**图表建议**：
1. RSI 仪表盘
2. 情绪分布图
3. 市场状态时间轴
4. 持仓主题热度图

#### 8.1.5 持仓关联个股分析

**功能说明**：围绕当前持仓个股和观察仓个股，提供 AI 解读摘要。

**分析对象**：
1. 当前股票持仓
2. 当前持仓的核心观察标的
3. 用户自选观察仓

**输出结构**：
1. 方向判断
2. 核心变化
3. 对组合的直接影响
4. 真正受益或受损环节
5. 市场是否误判
6. 需要继续跟踪的信号
7. 原文依据链接

**输出模板**：
- 方向：偏多 / 偏空 / 中性
- 核心变化：一句话说明最近最重要变化
- 对组合的直接影响：说明这件事会如何改变当前组合风险或机会
- 真正受益 / 受损：指出真正传导到利润池的环节
- 市场误判：说明市场现在可能看错了什么
- 需要跟踪：列出 2 到 4 个关键信号

**图表建议**：
1. 个股事件时间线
2. 个股对组合贡献图
3. 风险信号卡片
4. 相关新闻摘要卡

#### 8.1.6 对冲与防御

**功能说明**：本模块不涉及交易执行，仅输出风险防御建议。

**输出目标**：
1. 当前最大风险来源
2. 降低风险的方向性建议
3. 可替代持仓思路
4. 极端行情预案

**示例输出**：
- 当前风险集中于 AI 基础设施主线单一暴露。
- 若希望降低组合波动，可考虑降低高相关性仓位集中度。
- 若市场进入拥挤多头阶段，建议优先提高现金与防御性配置比例。

### 8.2 事件与预警

#### 8.2.1 功能目标

在风险发生前提醒用户，而不是在波动发生后再解释。

#### 8.2.2 子模块

1. 财报日历
2. 宏观事件提醒
3. 持仓异动预警
4. 风险阈值提醒

#### 8.2.3 预警规则示例

1. 单一持仓占比超过阈值
2. AI 主题暴露超过阈值
3. 持仓个股 24 小时内出现重大公告
4. 宏观高影响事件临近
5. 组合相关性显著上升

### 8.3 Telegram 集成

#### 8.3.1 功能目标

提供轻量触达和指令式查询能力。

#### 8.3.2 功能范围

1. 每日持仓分析日报
2. 风险预警推送
3. 指令查询账户信息

#### 8.3.3 指令示例

1. `/summary`：返回当前组合摘要
2. `/risk`：返回当前风险分析
3. `/positions`：返回当前持仓列表
4. `/cash`：返回现金与可用资金
5. `/events`：返回未来关键事件

### 8.4 MCP Server（只读）

#### 8.4.1 功能目标

把 `ibkr-dashboard` 作为本地只读 MCP 服务接入 Claude Desktop、Codex 等 AI 客户端，支持 AI 用结构化方式读取账户分析数据。这个方向与你草稿中的设想一致。

#### 8.4.2 设计原则

1. 仅提供只读工具
2. 默认本地使用
3. 返回结构化 JSON
4. 严格控制工具白名单

#### 8.4.3 建议工具清单

1. `get_portfolio_summary`
2. `get_positions`
3. `get_risk_exposure`
4. `get_position_news_summary`
5. `get_event_calendar`
6. `get_cashflow_summary`
7. `get_ai_concentration`

## 9. 数据与技术方案

### 9.1 核心数据优先级

#### P0 数据源

1. IBKR 账户报表 / Flex Query
2. 本地持仓计算结果
3. 通用行情数据
4. 财报与公告数据
5. 宏观事件日历

#### P1 可选数据源

1. 新闻情绪数据
2. 分析师 revision 数据
3. Futu Skills Hub 补充能力
4. Futu OpenD 行情补充能力

### 9.2 外部依赖策略

本期采用“核心能力不依赖 Futu OpenD”的策略。

具体原则：
1. 没有 Futu OpenD 时，系统仍可正常工作
2. Futu Skills Hub 作为散户情绪、市场新闻、评论情绪等增强层，建议保留接入能力
3. Futu OpenD 主要作为更完整行情与交易数据的可选数据入口，不作为本期前置依赖
4. 即使未来接入 Futu Skills / OpenD，也必须走 provider 抽象层
5. 所有市场分析信号都必须可替换、可降级

### 9.3 图表方案

图表统一采用 Apache ECharts。其官方说明支持 20 多种图表类型、多个组件任意组合，并支持 Canvas / SVG 渲染切换，适合复杂金融分析看板。

### 9.4 UI 风格方案

本期前端视觉重构采用 Nothing Style 方向，目标不是机械复制手机官网，而是提炼其可用于金融产品的设计特征：

1. 黑白灰主色与少量强调色
2. 强网格感与留白控制
3. 简洁、克制、工业感、轻未来感
4. 卡片层级清晰、边框明确、信息密度高但不压迫
5. 重点风险状态与预警信息用高对比方式突出

视觉落地要求：
1. 保持高可读性，避免仅为“设计感”牺牲金融信息效率
2. 图表、数字、标签必须优先服从分析与比较
3. 首页与持仓分析页风格统一

## 14. 风险与注意事项

1. 若市场分析过度依赖单一外部 provider，会造成系统耦合过高
2. 若 AI 输出未强制围绕持仓，容易重新退化为泛市场摘要工具
3. 若 Nothing Style 只学“样子”不学层级，会变成好看但难读的金融 UI
4. 若 MCP 工具边界不清，会增加只读系统的安全风险

## 15. 结论

IBKR Dashboard V2 本期明确聚焦“持仓分析”。

其中，市场分析与个股分析不独立存在，而是作为持仓分析中的解释层能力，为当前组合服务。

本期同时明确：

1. 首页沿用 V1，不重复建设
2. 期权内容延期到下一版
3. UI 风格采用 Nothing Style 方向重绘
4. Futu Skills Hub 继续保留接入价值，用于散户情绪、市场新闻等增强能力
5. Futu OpenD 仅作为可选行情补充层，不作为本期前置依赖

最终目标不是告诉用户“市场发生了什么”，而是告诉用户：

- 这些变化和你的组合有什么关系
- 你的风险正在如何变化
- 你接下来最应该关注什么
